slide 2 :



In this slide, I explain the current situation when a new version of the application is released.

Our automation framework is built for a specific version of the application, for example version X.

When a new version of the application is released, like version Y, the tests may no longer work.

This happens because some elements in the application change, like buttons or fields.

Because of that, we need to manually update the selectors in the test framework.

After updating the selectors, the framework becomes compatible with the new version of the application.


slide 3:


In this slide, I explain how we use AI to help update selectors.

We use our internal AI called Daisei.

We can access it in two ways: through a Python script using an API key, or directly in VS Code with the Continue extension.

The AI connects to different tools using MCP servers running in Docker.

One MCP server connects to GitLab, where our automation framework code is stored.
From there, the AI can read the current selectors used in the tests.

Another MCP server connects to Appium, which allows the AI to interact with mobile devices.
On these devices, we can install the new version of the application and extract the list of locators.

The AI then compares the locators from the application with the ones used in the framework.

Finally, it generates a file containing the updated locators for the new version.

This file still needs to be reviewed and integrated into the framework



slide 4:

After comparing the locators from the application and the ones in the framework, the AI generates a file containing the selectors for the new version of the application.

In this example, we can see a list of locators extracted from the conversation screen.

Each locator corresponds to an element in the user interface, such as text fields, buttons, or navigation items.

This file helps us quickly identify the selectors that may need to be updated in the framework.

After that, we review the file and integrate the necessary changes into the automation tests.




slide 5:

In this second use case, our goal is to improve our test documentation and convert our test cases into Gherkin format.

Today, many test cases are written manually by testers in Jira.

Using our internal AI, Daisei, we can read these test cases and generate a summary of the test scenario.

This summary helps the AI understand what the test is supposed to verify.

At the same time, the AI can also read the automation framework code stored in GitLab, to understand the existing test structure.

Based on this information, the AI generates a skeleton of the test in Gherkin format.

This skeleton can then be used by testers to quickly create or update automated tests.



slide 6:

The original test is written by manual testers in Jira and can sometimes contain a lot of information.

The AI reads this test case and generates a simplified summary of the scenario.

This summary helps to clearly identify the main goal of the test and the important steps.

By doing this, the AI can better understand the test before generating the automation code.

This step is important because it prepares the information that will be used to create the Gherkin test skeleton.





slide 7:

In this slide, we can see an example of the Gherkin skeleton generated by the AI.

Using the summary from the Jira test and the framework code, the AI generates the basic structure of the test.

The test is written in Gherkin format, with the usual structure: Given, When, and Then.

This skeleton gives us a good starting point to write the automated test.

Testers can then review the scenario and complete the implementation if needed.

This helps us save time and accelerate the creation of automated tests.


To conclude, these two use cases show how AI can help us reduce manual work and speed up both framework maintenance and test creation.
